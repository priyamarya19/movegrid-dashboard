MoveGrid Dashboard — Complete Tech Overview

A plain‑English guide to what the system is, how it's built, how it works, and why these technologies were chosen.

1. What this project is

MoveGrid Dashboard is the internal web application that runs an electric‑scooter rental business. It replaces spreadsheets and manual tracking with one organised system that manages:

  • Riders — the customers who rent scooters (onboarding, KYC documents, rent dues).
  • Vehicles — the scooter fleet (status, hub, owner, assignment history).
  • Hubs — the physical locations where scooters are parked and handed over.
  • Leads — prospective riders/investors in the sales pipeline.
  • Investors — people who fund scooters and receive monthly payouts, with their own self‑service portal.

It serves four types of users, and each only sees what's relevant to their job:

RoleWhat they manage
AdminEverything — full control, investors, audit logs, user accounts
Ops ManagerDay‑to‑day operations: riders, vehicles, hubs, leads
Hub InchargeTheir hub's riders and vehicles
InvestorOnly their own portfolio, profile, and support — a self‑service portal

So the app is really two things in one: a back‑office for staff, and a customer (investor) portal.

  • The browser shows the screens and is what users click around in.
  • The MoveGrid app is a single program (built with Next.js) that does both jobs: it builds the screens and runs the business logic on the server.
  • The database (PostgreSQL on AWS RDS) permanently stores all the data.
  • S3 stores uploaded files (photos, documents); SES sends emails.

A key idea: most apps split into a separate "frontend" and "backend" written in different languages. Here, one app and one language (TypeScript) does both, which keeps things simpler.

6. How it works — following one real request

The clearest way to understand it is to follow what happens when an investor opens "My Portfolio":

  1. The web address maps to a folder. /portfolio → app/portfolio. Next.js runs that page.
  2. A security gatekeeper runs first. Before the page loads, a check (the "middleware") reads the login token from the cookie. No valid token → redirected to login.
  3. The page builds itself on the server. This is a "Server Component" — it runs on the server. It runs SQL to fetch only this investor's vehicles, payouts and totals from PostgreSQL.
  4. The numbers are calculated — earnings, payouts remaining, ROI, and the environmental‑impact figures.
  5. A finished screen is sent to the browser — assembled on the server, so it appears quickly and already filled in.
  6. Interactive bits run in the browser. The "edit bank details" form or "request full number" popup are "Client Components" — they run in the browser to react instantly to clicks.
  7. Saving goes through an API endpoint. A submitted form is sent to a backend endpoint under app/api/..., which checks who you are, validates input, runs SQL to save, and replies. The screen refreshes.
One line: open a page → login is checked → the server fetches data with SQL → a ready‑made screen loads → forms save back through API endpoints.

Server‑rendered vs browser‑rendered: pages are built on the server so they load fast and securely (database access never leaves the server). Only the interactive parts run in the browser. We get the speed of server rendering and the slickness of an interactive app.

7. The database

  • PostgreSQL (on AWS RDS) is the single source of truth — everything is stored here.
  • Data is organised into schemas (labelled sections): one for operations (riders, vehicles, hubs, investors, payouts), one for logins/users, one for leads, one for audit logs.
  • Main tables include: riders, vehicles, hubs, rider_vehicle_assignments (who has which scooter), rider_payments, investor_profiles, investor_payouts, leads, users+roles, audit_logs.
  • We write SQL directly (rather than an automatic translator called an "ORM"), for precise control and performance on heavier calculations — like who's overdue on rent, or totalling investor payouts.

8. Security & access control

  • Passwords are never stored as text — they're hashed with bcrypt, a one‑way scramble.
  • Logins use a secure token (JWT) stored in a protected cookie, re‑checked on every request, expiring after 8 hours.
  • A gatekeeper (middleware) blocks every page for anyone not logged in.
  • Role‑based access — each page and endpoint checks your role, so an investor can never reach admin data.
  • Sensitive data is masked — an investor sees a rider's phone as 98XXXXXX10 and Aadhaar with only the last 4 digits; the full number needs a logged, emailed request.
  • Bank‑detail changes require approval — an investor's edit is marked "pending" until an admin verifies it.
  • Files are served via short‑lived secure links — S3 documents aren't public; links expire in an hour.
  • Live and test data are kept in separate databases.

10. Environments & how we ship changes

There are two environments, deliberately separated:

  • UAT (testing) — dash‑uat.movegrid.in, uses the test database. Every change is tried here first.
  • Production (live) — dash.movegrid.in, uses the live database with real customers.

Both run on the same server but are told which database to use by one setting (RDS_ENV) configured per app in ecosystem.config.js — this is what keeps live and test data apart.

A deployment is: push the code → on the server pull it, install any new tools, run any database updates ("migrations"), rebuild, and restart with pm2. Changes are verified on UAT before production.

11. Why this technology — industry standard & easy to hire for

Every piece of this stack is a current industry standard with a huge community. That matters practically: more documentation, fewer dead ends, proven reliability, and — importantly — a large pool of developers who already know these tools, so hiring is easier, faster and cheaper.

  • JavaScript / TypeScript is the world's most widely used programming language, year after year — the biggest talent pool of any language. It runs both the screens and the server, so one developer can work across the whole app.
  • React is the most popular way to build web interfaces, and Next.js is its leading framework — used by Netflix, TikTok and Notion. Proven at very large scale, heavily documented, easy to find answers for.
  • PostgreSQL is the most popular and most‑loved open‑source database — extremely reliable, free, widely known.
  • AWS is the market‑leading cloud platform — a standard skill most infrastructure engineers already have.
  • Tailwind CSS is today's dominant styling approach, so the UI is quick to build and consistent.

Why it's the right call: the stack is modern (current best practice, not legacy), fast for users (server‑rendered), scalable (the same tools power some of the largest sites in the world), future‑proof (actively maintained, growing adoption), and easy to staff. Nothing here is niche or unusual that would make it hard to maintain or hand over.
